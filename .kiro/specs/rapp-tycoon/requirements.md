# Requirements Document

## Introduction

rApp Tycoon is a browser-based multiplayer strategy game where 2-6 players per session design, deploy, and manage simplified rApps that optimise a virtual 5G network. Players compete by reacting to dynamically generated network events (e.g., power outages, traffic spikes) by deploying and tuning rApps on their basestations. The game scores players on a combination of money, customer satisfaction, and network stability. The system is deployed on Kubernetes with a Java Spring Boot backend, React frontend, MySQL database, and a Python event-generation service. Real-time updates are delivered via WebSocket. The platform must support at least 100 concurrent players across multiple game sessions.

## Glossary

- **Game_Session**: A single game instance hosting 2-6 players with a defined start and end condition
- **Player**: A human user participating in a Game_Session, managing their own set of basestations
- **Basestation**: A virtual 5G network node owned by a Player that can experience events and have rApps deployed to it
- **rApp**: A simplified application that a Player deploys to a Basestation to optimise network metrics; has properties including Purpose, Cost, Benefit, Risk, Confidence, Deployment_Status, Version, Impact, and Side_Effects
- **Event**: A network incident generated by the Event_Generator that affects one or more Basestations (e.g., power outage, traffic spike)
- **Event_Generator**: A Python service that produces game events on a schedule or based on game state
- **Leaderboard**: A real-time ranking display showing all players' scores within a Game_Session
- **Network_Metrics**: Quantitative measures of network performance including health, customer experience, cost, energy efficiency, automation reliability, and SLA compliance
- **Lobby**: A waiting area where players gather before a Game_Session starts
- **Backend**: The Java Spring Boot service handling game logic, state management, and API endpoints
- **Frontend**: The React-based browser client providing the player interface
- **WebSocket_Service**: The real-time communication layer delivering updates to connected players
- **Score**: A composite value derived from money, customer satisfaction, and network stability used to rank players
- **Deployment_Status**: The lifecycle state of an rApp (e.g., deploying, active, disabled, rolling back)

## Requirements

### Requirement 1: Game Session Lifecycle

**User Story:** As a player, I want to create or join a game session through a lobby, so that I can compete with other players in a structured match.

#### Acceptance Criteria

1. WHEN a Player creates a new Game_Session, THE Backend SHALL create a Lobby that accepts 2-6 Players
2. WHEN a Player requests to join a Lobby, THE Backend SHALL add the Player to the Game_Session if the player count is below 6
3. IF a Player attempts to join a full Lobby, THEN THE Backend SHALL reject the join request with a descriptive error message
4. WHEN the Game_Session host starts the game, THE Backend SHALL verify at least 2 Players are in the Lobby and transition the Game_Session from lobby state to active state
5. IF the host attempts to start the game with fewer than 2 Players in the Lobby, THEN THE Backend SHALL reject the start request with a descriptive error message
6. WHEN the Game_Session end condition is met, THE Backend SHALL transition the Game_Session to completed state and display final scores
7. THE Backend SHALL persist all Game_Session state to the MySQL database

### Requirement 2: Player Basestation Management

**User Story:** As a player, I want to manage my own set of basestations, so that I can optimise my network performance and respond to events.

#### Acceptance Criteria

1. WHEN a Game_Session starts, THE Backend SHALL assign each Player a set of Basestations
2. THE Frontend SHALL display the Player's Basestations on a map view showing current status and active events
3. WHEN an Event affects a Basestation, THE Frontend SHALL visually indicate the affected Basestation within 2 seconds of the event occurring
4. THE Backend SHALL track Network_Metrics for each Basestation independently

### Requirement 3: rApp Deployment and Management

**User Story:** As a player, I want to deploy, disable, tune, and roll back rApps on my basestations, so that I can optimise my network in response to events.

#### Acceptance Criteria

1. WHEN a Player deploys an rApp to a Basestation, THE Backend SHALL validate the deployment request and update the rApp Deployment_Status to deploying
2. WHEN an rApp deployment completes, THE Backend SHALL apply the rApp's Impact to the target Basestation's Network_Metrics
3. WHEN a Player disables an rApp, THE Backend SHALL remove the rApp's Impact from the Basestation's Network_Metrics and set Deployment_Status to disabled
4. WHEN a Player tunes an rApp, THE Backend SHALL recalculate the rApp's Impact based on the new configuration
5. WHEN a Player rolls back an rApp, THE Backend SHALL revert the rApp to its previous Version and recalculate Network_Metrics
6. THE Backend SHALL enforce that each rApp deployment includes valid Purpose, Cost, Benefit, Risk, and Confidence values
7. IF an rApp combination causes a side effect, THEN THE Backend SHALL generate a side-effect Event and apply the negative Impact to affected Basestations

### Requirement 4: Event Generation

**User Story:** As a game designer, I want events to be generated dynamically during gameplay, so that players face challenges that require strategic rApp deployment.

#### Acceptance Criteria

1. WHILE a Game_Session is in active state, THE Event_Generator SHALL produce Events at configurable intervals
2. WHEN the Event_Generator produces an Event, THE Backend SHALL assign the Event to one or more Basestations and notify affected Players via WebSocket
3. THE Event_Generator SHALL support multiple event types including power outages, traffic spikes, hardware failures, and SLA breaches
4. IF an Event is not resolved within a configurable time window, THEN THE Backend SHALL apply escalating negative Impact to the affected Basestation's Network_Metrics

### Requirement 5: Scoring and Leaderboard

**User Story:** As a player, I want to see a real-time leaderboard showing all players' scores, so that I can track my competitive standing.

#### Acceptance Criteria

1. THE Backend SHALL calculate each Player's Score as a weighted combination of money, customer satisfaction, and network stability
2. WHEN a Player's Network_Metrics change, THE Backend SHALL recalculate that Player's Score
3. THE WebSocket_Service SHALL broadcast updated Leaderboard data to all Players in a Game_Session within 2 seconds of a Score change
4. THE Frontend SHALL display the Leaderboard showing each Player's rank, name, and Score components
5. THE Backend SHALL determine the winner of a Game_Session based on the highest composite Score at game end

### Requirement 6: Real-Time Communication

**User Story:** As a player, I want to receive real-time updates about game state changes, so that I can react promptly to events and see current standings.

#### Acceptance Criteria

1. WHEN a Player connects to a Game_Session, THE WebSocket_Service SHALL establish a persistent WebSocket connection
2. THE WebSocket_Service SHALL deliver Event notifications to affected Players within 2 seconds of Event generation
3. THE WebSocket_Service SHALL broadcast Leaderboard updates to all Players in a Game_Session when any Player's Score changes
4. IF a WebSocket connection is lost, THEN THE Frontend SHALL attempt reconnection with exponential backoff and resynchronise game state upon reconnection
5. THE Backend SHALL validate all incoming WebSocket messages using server-side validation rules

### Requirement 7: rApp Catalogue

**User Story:** As a player, I want to browse available rApps with their characteristics, so that I can make informed deployment decisions.

#### Acceptance Criteria

1. THE Backend SHALL provide a catalogue of available rApps including Energy Saver, Capacity Optimiser, Fault Predictor, SLA Guardian, Configuration Drift Detector, Traffic Balancer, and Alarm Noise Reducer
2. THE Frontend SHALL display each rApp's Purpose, Cost, Benefit, Risk, Confidence, and possible Side_Effects
3. WHEN a Player requests the rApp catalogue, THE Backend SHALL return the full list of available rApps with their current characteristics

### Requirement 8: Security and Input Validation

**User Story:** As a system administrator, I want the game to enforce server-side validation and access control, so that players cannot manipulate scores or access unauthorised sessions.

#### Acceptance Criteria

1. THE Backend SHALL validate all player action requests on the server side before applying state changes
2. THE Backend SHALL reject any direct modification attempts to Score or Network_Metrics values from client requests
3. WHEN a Player attempts to access a Game_Session they are not a member of, THE Backend SHALL deny the request and return an authorisation error
4. THE Backend SHALL validate all input data against expected types, ranges, and formats before processing
5. IF a request fails validation, THEN THE Backend SHALL return a descriptive error response without exposing internal system details

### Requirement 9: Scalability and Deployment

**User Story:** As a system operator, I want the platform deployed on Kubernetes with horizontal scaling, so that the system supports at least 100 concurrent players across multiple game sessions.

#### Acceptance Criteria

1. THE Backend SHALL be deployable as a Kubernetes pod with configurable resource requests and limits
2. THE Frontend SHALL be deployable as a Kubernetes pod that serves static assets and can scale horizontally
3. THE Backend SHALL support horizontal scaling to handle at least 100 concurrent Players across multiple Game_Sessions
4. THE Backend SHALL expose readiness and liveness probe endpoints for Kubernetes health checking
5. THE Backend SHALL use ConfigMaps for application configuration and Secrets for sensitive values

### Requirement 10: Network Metrics Simulation

**User Story:** As a player, I want network metrics to change over time based on my rApp deployments and active events, so that the game provides a realistic simulation experience.

#### Acceptance Criteria

1. WHILE a Game_Session is in active state, THE Backend SHALL update Network_Metrics for all Basestations at regular tick intervals
2. WHEN an rApp is active on a Basestation, THE Backend SHALL apply the rApp's Impact to that Basestation's Network_Metrics each tick
3. WHEN an Event is active on a Basestation, THE Backend SHALL apply the Event's negative Impact to that Basestation's Network_Metrics each tick
4. THE Backend SHALL calculate Network_Metrics including network health, customer experience, cost, energy efficiency, automation reliability, and SLA compliance
5. IF multiple rApps are active on a single Basestation, THEN THE Backend SHALL calculate combined Impact considering potential interaction effects

### Requirement 11: Docker and Service Containerisation

**User Story:** As a developer, I want all services containerised with Docker, so that the application can be built and deployed consistently across environments.

#### Acceptance Criteria

1. THE Backend SHALL include a Dockerfile that produces a runnable container image
2. THE Frontend SHALL include a Dockerfile that produces a runnable container image serving the React application
3. THE Event_Generator SHALL include a Dockerfile that produces a runnable container image for the Python service
4. THE project SHALL include a Docker Compose configuration that starts all services together for local development

### Requirement 12: Game State Persistence

**User Story:** As a player, I want my game progress saved reliably, so that game state is not lost during play.

#### Acceptance Criteria

1. THE Backend SHALL persist all Game_Session state changes to the MySQL database within 1 second of occurrence
2. THE Backend SHALL persist Player actions, rApp deployments, and Event outcomes to the MySQL database
3. WHEN a Player reconnects after a disconnection, THE Backend SHALL restore the Player's full game state from the database
4. THE Backend SHALL use database transactions to ensure consistency of related state changes
